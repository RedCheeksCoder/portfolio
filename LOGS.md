## 2026-07-18 — Replaced GHL booking iframe with a custom Vercel-backed booking widget

- **Why:** Bryan asked what could be done about the GHL calendar iframe not matching the site's branding. Confirmed the real constraint first (cross-origin iframe, Same-Origin Policy blocks page CSS from reaching in; verified against GHL's actual support docs that only Primary/Background color pickers exist, only on their Neo widget style, no custom-CSS hook — not guessed). Offered the GHL-dashboard color fix as a zero-code option, but Bryan asked to build a fully custom calendar instead that (1) matches the brand exactly, (2) checks his real Google Calendar availability, and (3) still creates GHL contacts/appointments so his existing reminder-email automations keep firing. This is a genuine architecture shift — the site went from zero-backend static HTML to static HTML + a Vercel serverless API, since Google/GHL credentials can't live in client-side code. Used full Plan Mode for this one (Explore was skipped — no existing backend to explore — but a Plan subagent designed the file structure/API/frontend approach before writing any code), given the size and the number of real design decisions involved (backend host, sync depth, GHL integration depth, replace-vs-coexist) were confirmed with Bryan via AskUserQuestion first.
- **New backend** (`api/` + `package.json`/`vercel.json`/`.env.example`/`.gitignore`, all at the `Portfolio/` root):
  - `api/_lib/config.js` — `BOOKING_CONFIG`: working hours (Mon–Fri 9–5 default), Asia/Manila timezone, 30-min slots, 10-min buffer, 12h minimum notice, 30-day advance window. **All agent defaults, not yet confirmed by Bryan** — flagged in `CLAUDE.md` §8.
  - `api/_lib/slots.js` — pure, dependency-free slot-generation + timezone math (uses `Intl.DateTimeFormat` to compute real IANA offsets rather than hardcoding +08:00, so it'd still be correct if the timezone ever changed to a DST-observing one). Unit-tested directly with `node` — 12 assertions covering weekday detection, full-day slot counts, exact ISO-offset formatting, busy-block+buffer exclusion (verified a 10-11am busy block with a 10min buffer correctly removes the 9:30, 10:00, 10:30, and 11:00 slots specifically), minimum-notice exclusion, and booking-window bounds. All passed.
  - `api/_lib/googleCalendar.js` — service-account JWT auth (`googleapis` package), Freebusy query for availability, `events.insert` for booking (`sendUpdates:'none'` — documented limitation, see below).
  - `api/_lib/ghl.js` — plain `fetch` calls to GHL's v2 API (`services.leadconnectorhq.com`) for contact upsert + appointment creation. Wrapped so GHL failures never throw — returns `{ok:false, error}` instead, since the plan calls for GHL failure to be non-fatal to the visitor (see below).
  - `api/availability.js` (`GET ?date=YYYY-MM-DD`) and `api/book.js` (`POST`) — the two actual HTTP endpoints, thin handlers over the `_lib` modules.
  - **Race-condition handling:** `book.js` re-checks the specific slot's availability immediately before writing, returning `409 slot_taken` if it's gone. This narrows but doesn't fully close the race window (documented as an accepted, tolerable limitation for solo-consultant booking volume, not a silent gap).
  - **Partial-failure handling:** if the Google Calendar write succeeds but the GHL sync fails, the visitor still sees a successful booking (`ghlSynced:false` in the response) since the calendar hold is real either way — the failure is only logged server-side for Bryan to catch manually. This was an explicit design decision (confirmed via the approved plan), not an oversight.
- **New frontend widget** (`#bookingWidget` replacing `.booking-frame`/`<iframe>` in `#book`): three-step flow (month calendar + slot picker → contact form → confirmation), styled entirely with the site's existing tokens — reuses `.field`/`.btn-primary` verbatim, new `.day-cell`/`.slot-btn` styled off the existing `.filter-btn` pill pattern. Vanilla JS, IIFE-wrapped, appended after the work-filter code in the trailing `<script>`. Also deleted the GHL `form_embed.js` script tag — **the site now has zero external JS dependencies.**
- **Verification:**
  - `slots.js` pure-logic unit tests (12/12 passed, see above).
  - All new `api/*.js` files syntax-checked with `node --check`.
  - Full frontend flow tested with Playwright against a **local HTTP server** (not `file://` — fetch() from a file:// origin doesn't reliably hit route interception, so a throwaway `http.createServer` was used to serve `index.html` on localhost) with `/api/availability` and `/api/book` mocked: calendar renders, date selection loads slots, slot selection shows the form, successful submission shows the confirmation with correctly localized time, and a mocked `409 slot_taken` response correctly routes back to the date step with an inline error and reloads that date's slots. Existing site features (work filters, case-study modals) re-verified working alongside the new widget with zero regressions. Screenshots taken of both widget steps — visually confirmed on-brand (dark background, orange accents, correct fonts).
  - **Not tested: anything against real Google/GHL APIs** — impossible without real credentials, which the agent cannot create (requires Bryan's own account logins).
- **Known limitations documented (not silently accepted):** soft race-guard (see above), no native Google Calendar invite email to visitors since a bare service account can't send them without Workspace domain-wide delegation (GHL's reminder automation is the real notification channel), and GHL-sync failure being non-fatal to the visitor response.
- **This feature is built but NOT live.** It's blocked entirely on Bryan completing: (1) a Google Cloud service-account setup + sharing his calendar with it, (2) a GHL Private Integration token + Location ID + Calendar ID, (3) Vercel deployment + env vars. Full checklists for all three are in `CLAUDE.md` §12. Until that's done, deploying this code as-is would mean a booking widget with no working backend behind it.
